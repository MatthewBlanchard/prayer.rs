# prayer.rs

A Rust runtime for SpaceMolt automation via a small command language.

Scripts are generated by an LLM from natural language, inspected and edited by the user, then executed deterministically by the runtime against the live SpaceMolt API. The script is always visible. Execution can be halted, edited, and resumed at any point.

```
Client (web / CLI / agent) -> prayer.rs -> SpaceMolt
```

---

## What it does

You describe what you want in plain language. The generation agent turns that into a PrayerLang script. The runtime executes it tick-by-tick, surfacing state as it goes.

```
"mine carbon ore at nexus then sell"
```

becomes:

```
go nexus;
mine carbon_ore;
sell;
```

The runtime then drives that script step-by-step against SpaceMolt, tracking multi-turn commands like `go` and `mine` across ticks, checkpointing state so execution can be paused and resumed.

---

## PrayerLang

PrayerLang is a command language, not a programming language. The distinction is intentional.

- One command per line, terminated by `;`
- `if <CONDITION> { ... }` — evaluate once on entry
- `until <CONDITION> { ... }` — repeat body while condition is false
- Variables injected at script-load time are immutable
- No mutable state, no functions, no side effects outside game commands

The narrow surface makes scripts easy for an LLM to generate correctly and easy for a human to read and approve before execution. The immutability means a script can be fully understood by reading it top-to-bottom — there is no state to track mentally.

### Commands

`mine` `survey` `explore` `go` `dock` `repair` `refuel` `sell` `buy` `cancel_buy` `cancel_sell` `retrieve` `stash` `accept_mission` `abandon_mission` `switch_ship` `install_mod` `uninstall_mod` `buy_ship` `buy_listed_ship` `commission_ship` `sell_ship` `list_ship_for_sale` `wait` `craft` `halt`

Command behavior note: `mine` and `mine <resource_id>` are multi-turn operations that continue mining until cargo is full, then the script advances to the next command.
Resource targeting note: `mine ore;` targets ore mining, but outputs can still include mixed/byproduct cargo rather than only ore.

### Conditions

| Token | Type | Description |
|---|---|---|
| `FUEL()` | Numeric | Current fuel percentage |
| `CREDITS()` | Numeric | Current credit balance |
| `CARGO_PCT()` | Numeric | Cargo hold fill percentage |
| `CARGO(item_id)` | Numeric | Units of item in cargo |
| `MINED(item_id)` | Numeric | Units mined this script run |
| `STASHED(item_id)` | Numeric | Units stashed this script run |
| `STASH(poi_id, item_id)` | Numeric | Units in storage at POI |
| `MISSION_COMPLETE(mission_id)` | Boolean | Whether mission is complete |

`MINED` and `STASHED` counters reset to zero when a new script is loaded.

### Macros

| Macro | Resolved | Value |
|---|---|---|
| `$here` | Script load time | Current system — fixed origin |
| `$home` | Per command emit | Current home base — may change |
| `$nearest_station` | Per command emit | Nearest station — changes as you travel |

## Execution pipeline

Scripts move through three stages before anything touches the game.

```
DSL source text
      │
      ▼
┌─────────────┐
│   PARSING   │  text → AstProgram
└──────┬──────┘  structural errors, unknown commands, wrong arg counts
       │
       ▼
┌─────────────┐
│   ANALYZER  │  AstProgram + GameState → AnalyzedProgram
└──────┬──────┘  macro resolution, arg canonicalization, fuzzy-match suggestions
       │
       ▼
┌─────────────┐
│  EXECUTION  │  emit one EngineCommand per tick
└─────────────┘  multi-turn state, error classification, checkpointing
```

Each stage has one responsibility. The analyzer validates a script against live game state before a single API call is made — unknown item IDs, bad destinations, and type mismatches are caught with suggestions ("did you mean `carbon_ore`?") rather than silent failures.

### Error classification

Execution failures are classified rather than swallowed:

- `Transient` — retry the same command next tick
- `Skip` — log and advance past this command
- `Fatal` — halt the script

### Checkpointing

Engine state is fully serializable. Multi-turn commands (`mine`, `go`, `refuel`, `explore`) store their continuation state in the execution frame, so the entire runtime state is part of every checkpoint automatically. Pause and resume work across process restarts.

## MCP and VFS

`prayer-mcp` is the MCP entry point for LLM clients. It connects to `prayer-api`, manages sessions/scripts, and exposes a deterministic virtual filesystem projection of runtime state so agents can inspect game context without calling SpaceMolt directly.

The VFS is session-scoped and queryable with MCP tools:

- `fs_ls` to list directories
- `fs_read` to read one virtual file
- `fs_query` for pipeline-style search (`find`, `grep`, `read`, `project`, `sort`, `unique`, `limit`)

High-signal paths currently include:

- `/status.json`
- `/context.json`
- `/missions/active.json` and `/missions/available.json`
- `/ship/ship.json` and `/ship/cargo.jsonl`
- `/market/station_market.json` and `/market/orders/*.jsonl`
- `/storage/items.jsonl`
- `/systems/index.json` and `/systems/{id}.json`
- `/catalog/**` and `/exploration/**`
- `/station/context.json`

The full mapping target and current implementation notes are documented in:

- `docs/game-state-vfs-mapping.md`
- `docs/prayer-mcp-virtual-filesystem-plan.md`

## API

Language reference: see [`LANGUAGE.md`](LANGUAGE.md).

### Service

```
GET  /health
```

### Sessions

```
GET    /api/runtime/sessions
POST   /api/runtime/sessions
POST   /api/runtime/sessions/register
GET    /api/runtime/sessions/:id
DELETE /api/runtime/sessions/:id
```

### State (long-poll supported)

```
GET /api/runtime/sessions/:id/state?since=<version>&wait_ms=<timeout>
GET /api/runtime/sessions/:id/snapshot
GET /api/runtime/sessions/:id/status
GET /api/runtime/sessions/:id/route?target=<system_id>
```

Long-poll behavior: `200` when state advances, `204` on timeout with no change. Current version in `X-Prayer-State-Version` response header.

### Scripts

```
POST /api/runtime/sessions/:id/script
POST /api/runtime/sessions/:id/script/execute
POST /api/runtime/sessions/:id/halt
```

`execute` is blocking — it returns when the script halts or completes.

## Building and running

Requires Rust 1.78+.

```bash
cd prayer.rs
cargo build --release
cargo run -p prayer-api
```

Tests:

```bash
cargo test
```

### prayer-mcp-client quick commands

From `prayer.rs/`, you can run:

```bash
scripts/prayer-mcp-client-test.sh
```

That runs:
- `cargo test -p prayer-mcp-client`
- `cargo test -p prayer-mcp-client --test local_mcp_e2e -- --ignored`

For a one-command local smoke run (`prayer-api` + `prayer-mcp` + `prayer-mcp-client`):

```bash
scripts/prayer-mcp-client-smoke.sh \
  --llm-base-url http://127.0.0.1:11434/v1 \
  --model your-model
```

For an interactive terminal conversation (REPL/TUI-lite):

```bash
cargo run -p prayer-mcp-client -- chat \
  --repl \
  --llm-base-url https://api.openai.com/v1 \
  --model gpt-4o-mini \
  --api-key "$OPENAI_API_KEY"
```

REPL commands:
- `/exit` or `/quit` to leave
- `/clear` to reset conversation history

Simplest one-command launcher (boots API + MCP + REPL chat):

```bash
scripts/run-prayer-chat.sh
```

It uses:
- `OPENAI_API_KEY` from your environment
- model `gpt-4o-mini`
- base URL `https://api.openai.com/v1`

Environment:

```
SPACEMOLT_BASE_URL      SpaceMolt API base URL
LOCAL_AUTH_BYPASS=1     Skip SpaceMolt auth (local dev only)
```

## Client architecture notes

- One long-poll worker per session, re-issued immediately after each response
- Pass `since=<version>` from the `X-Prayer-State-Version` header back each request
- Keep command dispatch on a separate task — don't block controls on the poll loop
- Treat prayer.rs as source of truth for runtime state; surface error bodies directly

## Known limitations and bugs

**`MINED` / `STASHED` predicates always return 0** — predicates currently read per-turn delta fields that are never populated. The session accumulators inside the engine are correct but not yet wired into `GameState` before each tick. Scripts using `until MINED(carbon_ore) > 5` will not behave as expected.

**No authentication or authorization** — prayer.rs is scoped to trusted/internal deployments. Do not expose to untrusted networks without adding an auth layer.

**API pre-stable** — endpoint shapes and DTO fields may change. Pin client and server to the same commit.
